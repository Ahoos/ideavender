# 🚀 안티그래비티(Antigravity) 기반 바이브코딩 표준 프로젝트 세팅 가이드

이 문서는 **안티그래비티(Antigravity) AI 에이전트**를 활용해 외부 라이브러리나 빌드 도구 없이, 순수 바닐라(Vanilla) HTML/CSS/JS 환경에서 고품질의 웹 서비스를 제작하기 위한 프로젝트 구성 표준과 에이전트 규칙을 정의한 가이드라인입니다.

---

## 1. 표준 폴더 구조 (Folder Structure)

유지보수와 리팩토링의 편의성을 높이기 위해 프로젝트 폴더 내부에 역할을 철저히 구분하는 폴더 구조를 사용합니다. 안티그래비티 에이전트가 코드를 작성할 때도 이 구조를 반드시 준수하도록 합니다.

```text
📂 내-프로젝트-폴더/
├── 📄 index.html          # 앱의 시작점이 되는 메인 HTML 파일 (type="module" 적용)
├── 📂 css/                # 스타일시트(CSS) 저장 폴더
│   └── 📄 main.css        # CSS 변수를 활용한 전역 디자인 스타일 및 레이아웃
├── 📂 js/                 # 자바스크립트(JS) 소스코드 폴더 (ES Modules 방식 사용)
│   ├── 📄 app.js          # 전역 상태 관리 및 초기화를 실행하는 메인 비즈니스 로직 파일
│   ├── 📂 components/     # 화면을 조립하는 재사용 가능한 UI 컴포넌트 모듈들 (HTML/JS 템플릿 형태)
│   └── 📂 utils/          # 반복되는 기능/계산, API 호출 등을 모아둔 공통 헬퍼 유틸리티 함수
└── 📂 assets/             # 이미지, 아이콘, 폰트 등 정적 리소스 폴더
```

> [!TIP]
> **왜 폴더 구조를 철저히 나눌까요?**
> 단일 파일에 모든 코드를 작성하게 되면 AI가 전체 맥락을 잃고 엉뚱한 코드를 짜거나 버그를 생성하기 쉽습니다. 컴포넌트와 유틸리티로 적절히 분할된 구조를 유지할 때 안티그래비티 에이전트가 가장 정확하고 견고한 코드를 생성할 수 있습니다.

---

## 2. 안티그래비티 규칙(Rules) 설정

프로젝트 루트 폴더에 아래 규칙 파일을 생성하면, 안티그래비티 에이전트가 작업을 수행할 때 이를 읽고 정해진 규칙대로 코딩을 진행합니다.

### 2.1. `.antigravityrules` 파일 생성
프로젝트 폴더 최상위(Root)에 `.antigravityrules` 라는 이름의 파일을 만들고 아래의 내용을 그대로 붙여넣습니다.

```markdown
# 안티그래비티 바이브코딩 표준 개발 지침

## 1. 기술 제약 조건 (순수 바닐라 및 ES 모듈)
- 외부 자바스크립트 라이브러리(jQuery, React, Vue, Lodash, Axios 등) 및 외부 CSS 프레임워크(Tailwind, Bootstrap 등)를 절대 사용하지 마십시오.
- HTML5, CSS3, 순수 웹 API(Fetch API 등) 및 JavaScript(ES6+) 표준 스펙만을 사용합니다.
- HTML 스크립트 연결 시 `<script type="module" src="js/app.js"></script>`를 적용하고, 파일 간 데이터와 함수 교환은 명시적으로 `import` 및 `export` 문법을 사용하십시오.
- 복잡한 빌드 도구(Webpack, Vite, npm 등)는 사용하지 않습니다.

## 2. 파일 크기 제한 (500줄 규칙)
- 단일 파일의 길이는 주석을 포함하여 최대 500줄을 초과할 수 없습니다.
- 코드 라인이 400줄에 임박하면, 독립적인 기능이나 컴포넌트를 분리할 계획을 세우십시오.
- 500줄이 넘어가기 전에 반드시 적절한 모듈(components/ 또는 utils/)로 리팩토링하고 로직을 쪼개어 파일로 분산해야 합니다.

## 3. 모듈화 및 폴더 규칙
- UI 컴포넌트(Header, Modal, Card 등)는 HTML/JS 템플릿 형태로 만들어 `js/components/` 폴더 하위에 격리 및 파일화합니다.
- 날짜 변환, API 호출, 데이터 필터링 등의 유틸리티 로직은 `js/utils/` 폴더 하위로 관리합니다.
- 중복되는 코드를 절대 방치하지 말고 모듈화하여 재사용성을 극대화하십시오.

## 4. 아키텍처 및 코딩 표준 규칙
- **상태 관리(State Management)**: 
  - 상태 정보(예: 할 일 목록 데이터, 활성화된 필터 등)를 DOM(HTML 태그 속성 등)에 직접 임시 저장하여 꺼내 쓰지 마십시오.
  - 자바스크립트 내부에 단일 상태 객체(`const state = { ... }`)를 구축하고, 상태 변경이 일어날 때 렌더링 함수가 화면을 업데이트하도록 유도하십시오.
- **이벤트 위임(Event Delegation)**:
  - 리스트 아이템이나 버튼 등 동적으로 생성 및 삭제되는 요소들에 개별적으로 이벤트 리스너를 바인딩하지 마십시오.
  - 언제나 정적인 부모 요소에 이벤트를 걸고, 이벤트 위임 패턴을 활용해 동적 요소를 제어해야 합니다.
- **비동기 처리**:
  - 모든 비동기 작업(서버 통신, 지연 실행 등)은 `async / await` 패턴을 사용하며, 예외 처리를 누락하지 않도록 반드시 `try / catch` 구문을 동반해야 합니다.
- **스타일 및 마크업 표준**:
  - Layout 작성 시 구조적 의미가 드러나도록 HTML5 시맨틱 마크업(`<header>`, `<main>`, `<section>`, `<footer>` 등)을 필수 적용합니다.
  - 전역 색상 값, 폰트 패밀리, 기본 여백 등은 CSS 변수(`--primary-color`, `--spacing-md` 등)로 정의하여 스타일시트 전반에 걸쳐 일관되게 활용하도록 합니다.

## 5. 서비스 기획 프레임워크 규칙
- 서비스 기획 및 설명 문서를 작성할 때에는 반드시 다음 4가지 핵심 프레임을 준수하여 구조화해야 합니다.
  1. **유저 (Target User)**: 서비스의 명확한 대상 사용자 페르소나 및 세그먼트 정의
  2. **문제점 (Pain Point)**: 타겟 유저가 겪고 있는 핵심 불편 사항 및 요구 사항
  3. **해결 방안 (Solution)**:
     - 매체: 웹서비스 (순수 HTML/CSS/JS 바닐라 스택)
     - 주요 기능 (1): 핵심 기능 정의 및 UI/동작 흐름
     - 주요 기능 (2): 보조 기능 및 유저 가치
     - 주요 기능 (3): 사용자 편의성 및 데이터 관리 기능
  4. **기대효과 (Expected Effect)**: 서비스 도입을 통해 유저와 비즈니스가 얻을 수 있는 가치 및 정성/정량적 효과
```
